CLI & IPC reference.

Deckhand ships a deckhand binary that wraps the same operations the GUI runs. Underneath that, a Go sidecar speaks JSON-RPC 2.0. If you want to drive Deckhand from your own scripts.

deckhand CLI

Installed alongside the app. On macOS, ~/Applications/Deckhand.app/Contents/MacOS/deckhand; on Linux, beside the AppImage; on Windows, in the install dir. Symlink it into your $PATH for convenience.

Flags: --json for machine-readable output, --check <name> to run one check.

deckhand install

# Headless install, same flow as the wizard, driven by a config file.
deckhand install --profile phrozen-arco \
                 --host printer.local \
                 --user mks --password-stdin \
                 --plan plan.yaml \
                 --yes
# plan.yaml lists answers to each screen. See the HITL doc for examples.

deckhand snapshot / restore

deckhand snapshot --host printer.local \
                  --paths '~/printer_data,~/klippy_extras' \
                  --out ./snap.tar.gz

deckhand restore --host printer.local \
                 --archive ./snap.tar.gz \
                 --strategy side-by-side

deckhand profiles

deckhand profiles list                # show installed profile registry
deckhand profiles update              # pull latest from deckhand-profiles
deckhand profiles show phrozen-arco   # YAML to stdout

deckhand debug-bundle

# Collects logs + run-state, runs the redaction pipeline, shows you a
# preview, then writes the zip.
deckhand debug-bundle --session last --out ./bundle.zip --review

Sidecar IPC (JSON-RPC 2.0)

The Flutter UI talks to the sidecar over a length-prefixed JSON-RPC connection on a Unix socket (POSIX) or named pipe (Windows). Nothing stops you from talking to it directly. Useful for tests, automation, or replacing the UI.

Envelope

// Request
{ "jsonrpc": "2.0", "id": 7, "method": "disks.list", "params": {} }

// Response
{ "jsonrpc": "2.0", "id": 7, "result": [
  { "path": "/dev/disk2", "size": 8589934592, "removable": true,
    "model": "SD Card Reader" }
]}

// Notification (server → client, no id)
{ "jsonrpc": "2.0", "method": "step.progress",
  "params": { "step": "kalico_clone", "pct": 42, "line": "Cloning…" } }

Method catalog (selected)

Full schema with parameter shapes lives in docs/IPC.md. The sidecar also self-documents: deckhand-sidecar --help-rpc dumps the catalog as Markdown.

Driving it from a script

#!/usr/bin/env python3
import json, socket, struct

s = socket.socket(socket.AF_UNIX)
s.connect('/tmp/deckhand-sidecar.sock')

def call(method, params=None, _id=[0]):
    _id[0] += 1
    msg = json.dumps({'jsonrpc':'2.0','id':_id[0],
                      'method':method,'params':params or {}}).encode()
    s.sendall(struct.pack('>I', len(msg)) + msg)
    n = struct.unpack('>I', s.recv(4))[0]
    return json.loads(s.recv(n))

print(call('doctor.run'))
print(call('disks.list'))
Sidecar lifecycle

The UI supervises the sidecar. It auto-restarts on crash, and any active session resumes from its run-state. If you're launching the sidecar yourself for scripting, pass --no-supervise.
